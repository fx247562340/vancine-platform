# 开发日志 / Devlog

本目录按月记录 Vancine 项目的日常开发过程，**与 `CHANGELOG.md` 互补**。

## 与其它记录的区别

| 文件 | 角色 | 写入时机 |
|------|------|----------|
| `CHANGELOG.md` | 用户视角的版本发布说明 | 发布时 |
| `docs/reports/` | 阶段性技术报告、上线复查 | 重大节点 |
| `docs/devlog/` | **每天的实际开发记录** | **每次涉及开发的任务后** |
| `git log` | 提交粒度的代码变更 | 每次 commit |
| Claude memory | Claude 跨会话用的项目状态 | Claude 自动维护 |

## 文件组织

- 一个月一个文件：`YYYY-MM.md`
- 文件内按日期倒序追加（最新在最上）
- 每天一个 `## YYYY-MM-DD` 二级标题
- 同一天多个任务，每个任务用 `### 任务标题` 三级标题分隔

## 必须记录的内容

凡是涉及代码、配置、部署、调试、决策的工作，都要记一条 entry。包括：

1. **新功能 / 修复**：问题、根因、修复方案、影响范围、提交哈希
2. **本地验证 / 发布**：本地 Docker 验证结果、生产发布结果、版本号
3. **调试 / 排障**：症状、排查路径、最终结论
4. **架构 / 设计决策**：选了什么方案、拒绝了什么方案、为什么
5. **失败的尝试**：哪条路走不通、原因、改用什么方案
6. **目录 / 文档整理**：移动了什么、删了什么、加了什么规则
7. **用户反馈**：用户当时说了什么、做了什么调整

## 每条 entry 推荐模板

```markdown
### 简短任务标题

**问题 / 起因**：发生了什么、用户说了什么、为什么要做。

**根因 / 分析**：调查到什么、确认是什么原因（如果是排障类）。

**方案 / 操作**：实际改了什么、用什么方式做。

**验证**：本地 Docker 怎么验的、生产 `/api/status` 是否正常。

**结果**：提交哈希、版本号、发布状态。

**遗留 / 注意**：还没做完的、风险、需要后续注意的。
```

不是每条 entry 都要全部字段。轻量原则：5 分钟能写完。

## 引用规则

- 在 `CLAUDE.md` 中通过 `docs/devlog/YYYY-MM.md` 引用
- 在 entry 中需要时反向链接 `CHANGELOG.md` 版本、`docs/reports/` 报告、git commit、Claude 记忆文件

## 命名约定

- 月度文件：`2026-06.md`、`2026-07.md`
- 不要按周分，月粒度足够
- 历史月份归档保留，不要删
