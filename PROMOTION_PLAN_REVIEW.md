# Vancine 零预算推广方案审查

> 审查日期：2026-06-09  
> 适用条件：个人开发者、尽量不付费、产品接近上线

## 一、结论

原推广方案覆盖面很全，但更像团队市场部方案，不适合个人开发者执行。

当前最优策略不是同时运营十多个平台，而是：

1. 先修复会影响信任和转化的上线问题。
2. 聚焦一个明确卖点和一个目标用户群。
3. 用 2-3 篇高价值内容反复分发，而不是每天生产新内容。
4. 把指标从注册量和 GitHub Star 改为首次成功调用和首次充值。
5. 在验证出真实付费用户前，不投广告、不做高额推荐奖励。

建议的核心定位：

> 面向海外独立开发者，用一个 OpenAI-compatible API 调用中国的图片、视频和 3D 模型，可用美元支付，无需中国账号。

这个定位比“一个 API 调用所有 AI 模型”更有差异化。通用 AI API 聚合已经非常拥挤，而海外开发者获取中国多模态模型仍有明确门槛。

## 二、上线前必须处理的问题

以下问题不解决，推广带来的流量容易浪费，甚至引发负面评价。

### P0：立即处理

| 问题 | 当前情况 | 风险 | 建议 |
|---|---|---|---|
| 文档域名不可用 | `docs.vancine.com` 当前无法解析 | 用户无法完成 API 接入 | 发布前必须恢复，或暂时将文档链接指向可用的站内文档 |
| 宣传数据不真实 | About 页面声称已有数千开发者、50+ 国家、每日数百万请求、投资机构支持 | 对 API 服务的信任伤害极大 | 全部改为可验证的真实描述 |
| SOC2 等安全声明缺乏依据 | 页面使用 “SOC2-aligned”“企业级”等表述 | 可能产生合规和信誉风险 | 未完成对应措施前，改成具体功能描述 |
| 产品数量口径不一致 | 推广文案写 200+ 模型，公开定价接口当前返回 17 条记录 | 用户会质疑夸大宣传 | 只宣传当前真实可调用、测试通过的模型数量 |
| 免费试用承诺需确认 | 页面写有免费额度和 Get Started Free | 若注册后没有可用额度，会立即流失 | 确保新用户可以完成至少一次真实调用 |
| Sitemap 无效 | `/sitemap.xml` 当前返回普通首页 HTML | SEO 页面难以被正确发现 | 生成真实 sitemap，并提交搜索引擎 |

### P1：推广前一周完成

| 问题 | 当前情况 | 建议 |
|---|---|---|
| 没有完整转化漏斗 | Waitlist 会记录来源，但注册、首次调用、充值未形成统一来源分析 | 为每个渠道使用独立来源参数，至少记录访问、注册、首次调用、充值 |
| GitHub 定位混乱 | 仓库是 `QuantumNous/new-api` 的 fork，当前 0 Star，README 主要介绍上游项目 | 不要把 GitHub Star 当核心指标；先制作 Vancine 专属 README 和 3 分钟接入示例 |
| 两个域名内容重复 | `vancine.com` 与 `api.vancine.com` 当前均展示同一页面 | 明确主站、控制台和 API 的职责，设置 canonical，避免用户困惑 |
| Waitlist 与已上线产品冲突 | 产品似乎已可注册使用，但仍有 Early Access waitlist | 若已开放，CTA 应改成“免费试用”；若未开放，则所有推广统一指向 waitlist |
| 价格比较缺少证据 | 页面展示百分比节省和访问门槛 | 为每一项比较标注日期、规格和官方价格来源，并定期复核 |

## 三、对原推广方案的审查

### 可保留

- V2EX、Reddit、Hacker News 等开发者社区发布。
- 技术教程、真实价格对比、完整接入 Demo。
- 用户访谈和反馈收集。
- SEO 长期建设。
- 邀请机制，但奖励应在付费后触发。

### 应降级或删除

| 原计划 | 判断 | 原因 |
|---|---|---|
| 首周同时发 V2EX、掘金、知乎、小红书、X、Product Hunt | 删除 | 个人开发者无法高质量维护，且无法判断哪个渠道有效 |
| 小红书产品介绍图文 | 低优先级 | API 产品的精准开发者转化通常较弱，制作成本较高 |
| 创建微信与 Discord 两个社群 | 延后 | 没有活跃用户时，空社群会削弱信任并增加维护成本 |
| 联系 KOL 换评测 | 延后 | 产品和转化链路未验证前，曝光价值有限 |
| 推荐人 $10、被推荐人 $5、无上限 | 删除 | 容易被刷号，且个人开发者无法承担失控成本 |
| Google、X、掘金广告 | 删除 | 尚未验证获客成本和用户终身价值，不适合当前阶段 |
| GitHub Star 作为核心 KPI | 删除 | Star 不等于 API 用户，且当前仓库是上游项目 fork |
| 一个月 500 注册、30 付费用户 | 改为挑战目标 | 对零预算、零存量用户的个人项目偏乐观 |

## 四、推荐的零预算渠道

### 第一优先级：精准社区发布

投入：低到中  
见效：快  
可行性：高

只选择 2 个海外渠道和 1 个国内渠道：

- Reddit：围绕中国图片、视频、3D 模型的真实接入教程或价格测试。
- Hacker News Show HN：产品稳定、文档可用后发布一次。
- V2EX：以独立开发复盘、技术实现和真实数据为主，不写硬广。

社区帖应提供独立价值，即使读者不注册也能学到东西。推荐主题：

- 我测试了 10 个中国视频生成模型的 API：价格、速度和失败率
- 如何用 OpenAI SDK 调用 Seedance、Kling 和 Seedream
- 作为个人开发者，我为什么把 AI 网关从“全模型”缩到“中国多模态模型”

不要使用“省钱神器”“所有模型”“无风险”等广告式话术。

### 第二优先级：可搜索的技术内容

投入：中  
见效：慢  
可行性：高

只写 3 篇核心文章，并在不同平台调整开头后复用：

1. 接入教程：5 分钟完成第一次调用。
2. 真实对比：相同 prompt、相同规格下的价格、速度和效果。
3. 迁移指南：从 OpenAI SDK 或其他聚合 API 切换到 Vancine。

每篇文章只设置一个 CTA：

> 使用文中的完整示例，领取足够完成第一次调用的测试额度。

### 第三优先级：开源工具引流

投入：中  
见效：中长期  
可行性：中高

与其推广整个 fork 仓库，不如制作一个很小、很实用的独立仓库：

- `vancine-examples`：Python、Node.js、curl 示例。
- `china-ai-model-benchmark`：公开模型价格、延迟和输出对比。
- Dify、Cherry Studio、OpenWebUI 的 Vancine 配置模板。

小工具更容易获得精准 Star，也更容易自然链接到服务。

### 第四优先级：Build in Public

投入：低  
见效：慢  
可行性：中

每周发布一次真实进展，不需要每天发：

- 本周新增多少个真实调用用户。
- 哪个模型最受欢迎。
- 一次故障和修复过程。
- 用户提出的问题及产品改进。

真实的小数据比虚构的大数据更能建立信任。

## 五、推荐奖励的零预算设计

不要在注册时给推荐人固定 $10。建议改为：

> 被推荐用户完成首次充值后，双方各获得其首次充值金额的 10%，奖励上限 $3，仅可用于 API 调用。

优点：

- 没有真实收入就不会产生大额成本。
- 降低刷号套利风险。
- 奖励是 API 额度，边际成本低于现金。
- 能衡量推荐是否带来付费用户。

早期邀请制也可以人工执行：为前 20 名真正完成调用并反馈的用户赠送少量额度，换取一次 15 分钟访谈或公开案例授权。

## 六、两周零预算执行计划

### 第 1 周：修复转化链路

| 日期 | 任务 | 完成标准 |
|---|---|---|
| D1 | 修复文档域名、删除不真实宣传 | 所有公开声明均可验证 |
| D2 | 确认 5-10 个主推模型 | 每个模型都有成功调用记录、价格和限制说明 |
| D3 | 打通免费首次调用 | 新用户 10 分钟内可完成第一次请求 |
| D4 | 配置 Umami 或 GA，增加来源参数 | 能区分社区来源并追踪注册 |
| D5 | 完成 3 分钟快速开始和错误排查页 | 陌生开发者不需要咨询即可接入 |
| D6 | 制作一个真实模型对比页面 | 包含日期、规格、结果和方法 |
| D7 | 邀请 5 名开发者封闭测试 | 至少 3 人成功调用并给出反馈 |

### 第 2 周：小规模发布并验证

| 日期 | 任务 | 完成标准 |
|---|---|---|
| D8 | 发布一篇深度教程 | 只发布到最匹配的一个社区 |
| D9 | 回复评论、修复阻塞问题 | 当天响应所有有效问题 |
| D10 | 根据首篇内容制作短 Demo | 复用内容，不重新策划主题 |
| D11 | 发布第二个社区版本 | 调整标题和开头，避免机械复制 |
| D12 | 访谈成功调用与未成功调用用户 | 找出前三个转化阻力 |
| D13 | 更新首页和文档 | 用真实反馈改善文案 |
| D14 | 复盘漏斗，决定是否继续扩大渠道 | 根据数据决定下一轮 |

Product Hunt 和 Show HN 应放在文档、稳定性、首次调用体验都验证完成之后，而不是固定在第二周发布。

## 七、现实指标

首月不建议以收入和注册总量作为唯一目标。使用以下漏斗：

| 指标 | 保底 | 合格 | 优秀 |
|---|---:|---:|---:|
| 精准访客 | 100 | 300 | 1,000 |
| 注册或 Waitlist | 10 | 40 | 120 |
| 首次成功调用 | 5 | 20 | 60 |
| 7 日内再次调用 | 2 | 8 | 25 |
| 首次充值 | 1 | 5 | 15 |
| 完成用户访谈 | 5 | 10 | 15 |

核心指标建议设为：

> 每周新增的首次成功调用用户数，以及这些用户中 7 日内再次调用的比例。

如果有访问但没有注册，问题通常在定位或信任；有注册但没有调用，问题通常在文档、免费额度或接入体验；有调用但没有复用和充值，问题通常在模型质量、稳定性、价格或真实需求。

## 八、方向可行性分析

| 方向 | 可行性 | 判断 |
|---|---|---|
| 面向国内用户做通用 AI API 聚合 | 中低 | 竞争激烈，差异化不足，价格容易被比较 |
| 面向海外用户提供中国多模态模型 API | 高 | 有明确的账号、语言、支付和接口统一价值 |
| 主打所有 LLM 更便宜 | 低 | 容易陷入价格战，利润薄且用户忠诚度低 |
| 主打图片、视频、3D 的统一 API | 高 | 用户痛点更具体，内容展示效果好，客单价也更容易提高 |
| 依靠 Product Hunt 一次性爆发 | 低 | 发布日流量不等于持续使用，需已有用户配合 |
| 依靠教程、示例和真实 benchmark 长期获客 | 高 | 符合个人开发者能力，内容可以持续复用 |
| 免费额度换真实反馈 | 高 | 成本可控，能直接改进产品 |
| 无上限推荐奖励 | 低 | 容易套利，成本不可控 |

## 九、最终建议

未来 30 天只做三件事：

1. 把所有公开文案改成真实、可验证的陈述。
2. 围绕“中国图片、视频、3D 模型的全球 API 接入”制作一个优秀教程和一个公开 benchmark。
3. 手动帮助前 20 个开发者完成第一次调用，并记录他们为何留下或离开。

在出现至少 5 个真实付费用户前，不投广告、不联系大 KOL、不同时运营多个社群。当前阶段最稀缺的不是曝光，而是对“谁会持续付费、为什么付费”的确定性。
