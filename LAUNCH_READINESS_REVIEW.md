# Vancine 上线准备度审查

> 审查日期：2026-06-09  
> 审查范围：本地代码、构建测试、生产服务器、公开页面、支付、基础安全与运维

## 结论

**当前结论：NO-GO，不具备公开上线和扩大推广条件。**

当前版本可以继续用于受控的个人测试，但在完成 P0 阻断项前，不建议开放注册、公开推广或继续收款。

最严重的问题不是功能缺失，而是存在可直接影响管理员权限和支付余额的安全风险。

## P0：上线阻断项

### 1. 生产密钥已经公开

公开仓库的 `docker-compose.yml` 包含固定的数据库密码、Redis 密码和 `SESSION_SECRET`，生产服务器当前正在使用相同值。

影响：

- `SESSION_SECRET` 用于签名客户端 Session Cookie。
- Session 中直接保存并信任用户 `id`、`username`、`role` 和 `status`。
- 已知该密钥的攻击者可能伪造管理员 Session。
- 数据库和 Redis 密码也必须视为已经泄露。

必须处理：

1. 立即暂停公开注册和推广。
2. 生成新的随机 `SESSION_SECRET` 并重启服务，使现有 Session 全部失效。
3. 轮换 PostgreSQL 和 Redis 密码。
4. 从 Compose 文件中移除真实默认值，改用服务器端 `.env` 或 Secret 管理。
5. 即使清理 Git 历史，旧密钥仍应永久视为失效。

### 2. PayPal Webhook 可被伪造

`controller/topup_paypal.go` 当前从请求头提供的任意 `certURL` 下载证书，然后直接使用该证书的公钥验证签名。

当前实现没有：

- 限制证书 URL 必须属于 PayPal 官方域名。
- 验证证书信任链、有效期和用途。
- 调用 PayPal 官方 `verify-webhook-signature` 接口。

攻击者理论上可以使用自己的证书签署伪造的 `PAYMENT.CAPTURE.COMPLETED` 事件，并为自己创建的待支付订单充值。

必须处理：

1. 修复前关闭 PayPal 收款。
2. 使用 PayPal 官方 Webhook 验签接口，或完整验证官方证书链与域名。
3. 验证支付订单 ID、金额、币种、捕获状态与本地待支付订单完全一致。
4. 添加 PayPal Webhook、重复通知、伪造通知、金额不一致测试。

### 3. 应用端口直接暴露公网

生产服务器的 `3000` 端口当前可以从公网直接访问，绕过 Nginx 和 HTTPS。

必须处理：

- Compose 端口改为 `127.0.0.1:3000:3000`。
- 启用防火墙，仅开放 `22`、`80`、`443`。
- 禁用 SSH 密码登录和 Root 密码登录。

### 4. 没有生产数据库备份

服务器未发现有效数据库备份，也没有 Root Cron 备份任务。

必须处理：

- 每日自动执行 PostgreSQL 备份。
- 将备份同步到另一台机器或对象存储。
- 设置保留周期，并实际执行一次恢复演练。

### 5. 注册开放但没有反滥用保护

生产环境当前：

- 注册开放。
- 密码注册开放。
- 邮箱验证关闭。
- Turnstile 关闭。
- 新用户自动赠送 `$1` API 额度。

这会导致免费额度被自动化批量注册消耗。

必须处理：

- 开启邮箱验证和 Turnstile，或改为邀请码封闭测试。
- 对同 IP、邮箱域名、设备和调用频率增加限制。

### 6. 用户协议和隐私政策为空

公开状态接口显示用户协议和隐私政策均未启用，对应 API 返回空内容。

在处理用户数据并接受付款前，至少需要提供：

- 用户协议。
- 隐私政策。
- 退款与争议处理规则。
- 数据保存、上游传输和日志记录说明。

## P1：发布质量阻断项

### 自动化质量门禁失败

| 检查 | 结果 |
|---|---|
| `go test ./...` | 失败：Controller 1 项、Claude 转换 3 项、Stream Scanner 1 项 |
| `go vet ./...` | 失败：锁复制与多处不可达代码 |
| Default 前端 `build:check` | 失败：5 个 TypeScript 错误 |
| Default 前端普通构建 | 通过 |
| Default 前端 ESLint | 工具依赖错误，无法运行 |
| Classic 前端构建 | 通过，但存在 CSS 语法与超大 Bundle 警告 |
| Classic 前端格式检查 | 失败：149 个文件 |
| `docker compose config` | 通过 |
| `docker build -f Dockerfile.deploy` | 失败：Docker 构建上下文排除了两个前端 `dist` 目录 |

公开发布前应确保关键测试、类型检查和生产镜像构建均能稳定通过。

### 部署流程不可复现

- GitHub `deploy.yml` 执行 `docker compose up -d --build`。
- Compose 仅声明现有镜像，没有 `build` 配置，因此该流程不会构建或拉取新代码镜像。
- 项目记录明确说明生产服务器内存不足，不能在服务器上构建。
- 当前工作区存在大量未提交改动与未跟踪上线文档。

建议建立唯一、可重复、可回滚的部署流程：

1. CI 执行测试和前端构建。
2. CI 构建带不可变版本标签的镜像。
3. 生产服务器仅拉取指定版本镜像。
4. 部署前自动备份，部署后执行健康检查。
5. 失败时自动回滚上一镜像。

### 模型可用性测试不可复现

- `test_models.sh` 中的 API Key 当前无效。
- 图片测试使用 `512x512`，低于生产模型要求，必然失败。
- 视频测试使用不存在的 `/v1/videos/generations`，实际端点为 `/v1/videos`。
- 3D 测试和站内文档宣传 `/v1/3d/generations`，但该端点当前返回 `404`。
- 生产日志显示多个模型近期仍存在错误，尤其是 TTS 和部分图片模型。

公开推广前应建立自动化生产 Smoke Test，并只宣传通过测试的模型和端点。

## P2：建议尽快完成

### 基础设施与安全

- 磁盘使用率已达到 `80%`。
- Docker 镜像占用约 `31.6 GB`，构建缓存约 `11 GB`。
- UFW 未启用。
- Fail2ban 未启用。
- SSH 允许 Root 登录和密码登录。
- 未设置 HSTS、CSP、X-Frame-Options 等常见安全响应头。
- CORS 当前返回 `Access-Control-Allow-Origin: *` 与凭证允许配置，建议按实际域名收紧。

### 支付与订单

- PayPal 生产模式已启用。
- 数据库中有 9 笔成功订单和 7 笔长期 Pending 订单。
- 9 笔成功订单中有 7 笔缺少 Transaction ID，属于旧流程历史数据。
- 尚未实现 PayPal 退款、撤销和争议事件处理。
- 尚未发现自动清理过期待支付订单的逻辑。

### SEO 与运营

- `/sitemap.xml` 当前返回普通 HTML 首页，不是真实 Sitemap。
- 主站和 API 子域名展示重复前端内容。
- 尚未形成访问、注册、首次调用、充值的完整来源漏斗。

## 已通过项目

- `vancine.com` 与 `vancine.com` 可通过 HTTPS 访问。
- TLS 1.0 和 1.1 已禁用，TLS 1.2 和 1.3 可用。
- Nginx 配置检查通过。
- 生产主容器当前为 Healthy。
- PostgreSQL 与 Redis 未直接暴露公网端口。
- 未认证访问模型 API 会正确返回 `401`。
- `/docs`、`/pricing`、注册等公开路由可访问。
- 生产公开定价接口当前返回 17 条记录。
- 新用户免费额度设置为 `$1`。
- 本地 Default 与 Classic 前端普通生产构建均可完成。

## 建议上线顺序

### 阶段 A：立即止血

1. 关闭 PayPal。
2. 关闭公开注册，改为邀请码内测。
3. 轮换 Session、数据库与 Redis 密钥。
4. 将 `3000` 端口绑定至本机。
5. 建立并验证数据库备份。

### 阶段 B：修复质量门禁

1. 修复 PayPal Webhook 验签与测试。
2. 修复全部关键 Go 测试和 TypeScript 错误。
3. 修复 Docker 镜像构建与部署流程。
4. 修正文档、测试脚本和真实端点不一致问题。
5. 对每个主推模型连续执行生产 Smoke Test。

### 阶段 C：封闭测试

满足以下条件后，可邀请 5-20 名开发者封闭测试：

- P0 全部完成。
- 备份与恢复验证通过。
- 注册具备反滥用保护。
- 文档可帮助陌生用户完成首次调用。
- 主推模型连续 7 天无阻塞性故障。
- 小额真实 PayPal 支付、重复通知和退款流程均验证通过。

### 阶段 D：公开上线

封闭测试至少有 5 名真实用户完成首次调用，且没有未解决的 P0/P1 问题后，再开始公开推广。
